# 可用性設計：SLA/SLO、冗長化、フェイルオーバー、グレースフルデグラデーション

## 概要

「動くシステム」と「壊れにくいシステム」は別物。AI時代に評価されるのは、**何が壊れたら何が起きるかを事前に設計し、ユーザー影響を最小化できる人**。
可用性設計は「100%を目指す」ことではなく、**ビジネス要件に合った可用性目標を決め、コストとトレードオフを納得して選ぶ**こと。SLO を起点に冗長化・フェイルオーバー・劣化動作を設計する。

## 仕組みの要点

- **SLA**: 顧客との契約上の約束（破ると返金など）
- **SLO**: 自分たちが守る目標値（例: 99.9%）
- **SLI**: 実測値（成功率、レイテンシなど）
- **エラーバジェット**: `100% - SLO`。これを使い切るまでは新機能リリース可、超えたら安定化に集中
- **可用性の階層**: 99.9%（年8.76h停止）→ 99.99%（年52分）→ 99.999%（年5分）。1桁上げるごとにコストは桁違い
- **冗長化の単位**: インスタンス → ゾーン → リージョン。広いほど高可用だが遅延・コスト増
- **障害の連鎖を断つ**: タイムアウト・リトライ・サーキットブレーカー・バルクヘッド

## アンチパターン vs 正しい設計

| アンチパターン | 正しい設計 |
|---|---|
| 「とりあえず99.99%」と決める | ビジネス影響から逆算してSLOを決める |
| 全機能を同じ可用性で扱う | クリティカルパス（決済等）と非クリティカル（推薦等）を分離 |
| 障害時に500を返して終わり | 劣化動作（キャッシュ・デフォルト値）でコア機能を維持 |
| リトライを無限に | 指数バックオフ + 上限 + サーキットブレーカー |
| 冗長化したつもり（同一AZ内で多重化） | マルチAZ／マルチリージョン、依存先も含めて検証 |

## 設計例：FastAPI + Neon + Cloud Run

```python
# クリティカルパスは劣化動作で守る
from circuitbreaker import circuit
from fastapi import FastAPI
import redis.asyncio as redis

app = FastAPI()
cache = redis.from_url("redis://...")

@circuit(failure_threshold=5, recovery_timeout=30)
async def fetch_recommendations(user_id: str):
    # 外部APIや重いクエリ
    return await rec_service.get(user_id, timeout=0.5)

@app.get("/home")
async def home(user_id: str):
    user = await get_user(user_id)  # 必須（失敗したら500）
    try:
        recs = await fetch_recommendations(user_id)
    except Exception:
        recs = await cache.get(f"recs:{user_id}") or []  # 劣化
    return {"user": user, "recs": recs}
```

Cloud Run側: `min-instances=1`（コールドスタート回避）、複数リージョン + Cloud Load Balancer でフェイルオーバー。Neon は read replica で読み取り分散。

## トレードオフ

| 選択肢 | メリット | デメリット |
|---|---|---|
| シングルリージョン | 低コスト、低遅延、シンプル | リージョン障害で全停止 |
| マルチリージョン Active-Passive | 障害時切替可、コスト中 | RTO/RPO が発生、切替手順が必要 |
| マルチリージョン Active-Active | 無停止、最高可用 | データ整合性が難、コスト高 |
| 同期レプリケーション | データロスなし | 書き込み遅延増 |
| 非同期レプリケーション | 書き込み高速 | フェイルオーバー時にデータロス可 |
