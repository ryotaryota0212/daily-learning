# データベース設計のトレードオフ（正規化・インデックス・パーティショニング）

## 概要

- DBはアプリケーションの「最も剥がしにくい層」であり、初期設計の負債が最も長く残る
- AI時代でも「どこにデータを置き、どう読み書きするか」の判断はエンジニアの責務
- 正規化・インデックス・パーティショニングは**読み込み速度・書き込み速度・整合性・運用コスト**のトレードオフを決める3本柱
- Neon (PostgreSQL) + FastAPI + Cloud Run スタックを前提に「壊れにくい設計」を整理する

---

## 1. 正規化 vs 非正規化

### 仕組みの要点

- **正規化**: データ重複を排除しテーブルを分割。更新時の整合性が強い／JOIN コストが増える
- **非正規化**: 重複を許容して単一テーブルや JSONB に集約。読み込みが速い／更新時の不整合リスク
- 多くの実装は「3NFまで正規化 → 読み込みボトルネックの箇所だけ意図的に非正規化」が現実解
- PostgreSQL では `JSONB` カラムが「正規化と非正規化の中間」として強力（GIN index で検索可能）

### アンチパターン vs 正しい設計

| アンチパターン | 正しい設計 |
|---|---|
| 最初から JSON ですべて持って後で困る | 検索キーになるカラムは正規化、ペイロードのみ JSONB |
| 集計のたびに 5 テーブル JOIN | 集計結果はマテリアライズドビュー or サマリーテーブル |
| 「将来必要になるかも」で過度に分割 | 1:1 で常に JOIN する関係は1テーブルに統合 |

---

## 2. インデックス戦略

### 仕組みの要点

- インデックスは**読み込み高速化と引き換えに、書き込み・ストレージ・VACUUMコストを増やす**
- B-tree は等価/範囲検索、GIN は全文検索/JSONB、BRIN は時系列の巨大テーブル向け
- 複合インデックスは**WHERE/ORDER BY の左端から使う列順**が重要（カーディナリティが高い列を先頭に）
- カバリングインデックス（`INCLUDE`）で Index Only Scan を狙うとI/Oが激減
- 不要インデックスは `pg_stat_user_indexes` の `idx_scan = 0` で検出して削除する

### コード例（PostgreSQL / FastAPI 前提）

```sql
-- 複合 + カバリングインデックス: ユーザーの最新注文取得を最適化
CREATE INDEX idx_orders_user_created
  ON orders (user_id, created_at DESC)
  INCLUDE (status, total_amount);

-- JSONB の特定キー検索のみ高速化（GIN全体より軽量）
CREATE INDEX idx_orders_meta_source
  ON orders ((meta->>'source'));

-- 部分インデックス: アクティブ行のみ対象でサイズを縮小
CREATE INDEX idx_orders_pending
  ON orders (created_at)
  WHERE status = 'pending';
```

## 3. パーティショニング

### 仕組みの要点

- 1テーブルが**数千万行〜億行**を超え、削除や集計が重くなったら検討
- PostgreSQL のネイティブ宣言的パーティショニング（RANGE / LIST / HASH）が標準
- 時系列データは **RANGE by month/day** が定石。古い月を `DETACH` で一瞬で削除可能
- パーティションキーは**ほぼ全クエリのWHEREに含まれる列**を選ぶ（プルーニングが効く）
- アンチパターン: 「いつか巨大化するかも」で早すぎる分割 → 運用負担だけ増える

### 設計例

```sql
CREATE TABLE events (
  id BIGSERIAL,
  user_id BIGINT NOT NULL,
  occurred_at TIMESTAMPTZ NOT NULL,
  payload JSONB
) PARTITION BY RANGE (occurred_at);

CREATE TABLE events_2026_04 PARTITION OF events
  FOR VALUES FROM ('2026-04-01') TO ('2026-05-01');

-- 古い月を即時アーカイブ
ALTER TABLE events DETACH PARTITION events_2025_04;
```

## トレードオフ早見表

| 手段 | 読み込み | 書き込み | 整合性 | 運用コスト | 適用タイミング |
|---|---|---|---|---|---|
| 正規化を進める | 遅い (JOIN) | 速い | 強い | 低 | 通常のOLTP |
| 非正規化 / JSONB | 速い | 遅い (重複更新) | 弱い | 中 | 集約読み込み・スキーマ流動的 |
| インデックス追加 | 速い | 遅い | 不変 | 中 (肥大化) | 既存クエリのボトルネック解消後 |
| パーティショニング | 速い (プルーニング前提) | やや遅い | 不変 | 高 (DDL運用) | 行数が億単位 / 削除が多い |
| マテビュー | 非常に速い | 別途リフレッシュ | 鮮度に遅延 | 中 | 集計・ダッシュボード |

## Neon / Cloud Run 文脈での注意

- Neon は**スケールトゥゼロ**するため、コールドスタート時にコネクションプールが空 → `pgbouncer` (Neon の Pooler) 経由で接続必須
- Cloud Run は**インスタンスごとに独立したプール**を持つ。最大同時接続数 = `インスタンス数 × プールサイズ` が Neon の上限を超えないこと
- インデックス追加は `CREATE INDEX CONCURRENTLY` で本番ロックを避ける（トランザクション外で実行）
- パーティション切替は `pg_cron` 等で月次自動化。手動運用はミスの温床

## チェックリスト

- [ ] そのテーブルの**主要クエリ3つ**を列挙し、それぞれの実行計画 (`EXPLAIN ANALYZE`) を確認したか
- [ ] 書き込み頻度の高い列に**過剰なインデックス**を貼っていないか（不要インデックスは削除）
- [ ] JSONB に入れているデータのうち、**WHEREやORDER BYで使う列**は通常カラムに昇格したか
- [ ] 1億行を超える見込みのテーブルに**パーティション戦略**があるか（なければ後付けは高コスト）
- [ ] スキーマ変更は `CONCURRENTLY` / 短時間ロック / バックフィル戦略を含めて**ダウンタイムなし**で実行できるか

## 設計者のマインドセット

- 「とりあえず正規化」ではなく「**このクエリパターンならどの形が一番素直か**」から逆算する
- インデックスは"足す"より"残す価値があるか"を定期的に問う
- パーティションは**早すぎる最適化**になりやすい。指標（行数・削除頻度・クエリレイテンシ）が悪化してから入れる
- 設計判断は ADR に「なぜこの形にしたか・代替案・前提が崩れたら見直す条件」を残す
