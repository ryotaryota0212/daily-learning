# X上での効果的な開発過程の発信方法（ビルドインパブリック）

## 概要

ビルドインパブリック（Build in Public）は、開発の裏側を公開しながらファンとユーザーを同時に獲得する手法。完成品の宣伝より「作っている過程」を見せる方がエンゲージメントが高い理由は単純で、人は完成品より「未完成のストーリー」に感情移入するから。個人開発者はビルドインパブリックと最も相性が良いポジションにいる。

## 具体的な戦術

### 1. 週次ルーティンを確立する

毎日投稿は疲弊する。週に決まった「型」を作ることで持続可能にする。

**おすすめの週次パターン**:
- **月曜**: 今週やることリスト（WIP宣言）
- **水曜**: 途中経過のスクリーンショット or 動画
- **金曜**: 今週の振り返り + 数字の公開
- **随時**: バグ・失敗・学びのリアルタイム投稿

**1投稿の理想的な構成**:
1. 何をやったか（1行）
2. スクショ or 動画（必須）
3. 学び or 気づき（1〜2行）
4. ハッシュタグ（#個人開発 #buildinpublic）

### 2. 「失敗」を意図的に発信する

成功報告よりも失敗報告の方がエンゲージメントが2〜3倍高い。

**効果的な失敗投稿のパターン**:
- 「◯◯を試したけど完全に失敗した。理由は△△」
- 「今月の売上が先月比-30%。原因を分析した結果...」
- 「3日かけた機能をユーザーの一言で全部作り直し」

**なぜ効くのか**: 失敗を正直に話せる人は信頼される。また、同じ失敗をしそうな人が「参考になった」と保存・共有する。成功は妬みを生むが、失敗は共感を生む。

### 3. 数字の公開を習慣にする

数字は最も強力なビルドインパブリックのコンテンツ。

**公開すべき数字**:
- MRR（月間経常収益）の推移
- ユーザー数（無料/有料の内訳）
- 転換率、解約率
- PV、DAU/MAU
- 開発にかかった時間・コスト

**公開のフォーマット例**:
```
📊 4月の振り返り

MRR: ¥48,000（+15%）
有料ユーザー: 32人（+5）
解約: 2人
新機能: ダッシュボード改善
開発時間: 42時間

5月の目標: MRR ¥60,000
#個人開発 #売上公開
```

### 4. Before/After形式を活用する

改善のビジュアルは最もバズりやすいコンテンツ形式の一つ。

- UI改善のBefore/After
- パフォーマンス改善（レスポンス速度3秒→0.2秒）
- リファクタリングのコード比較
- 売上グラフの推移

**ポイント**: 2枚の画像を並べるだけで投稿が完成する。テキストは最小限でOK。

### 5. 意思決定の過程を見せる

「何を作ったか」より「なぜそう決めたか」の方が学びが大きく、保存・共有されやすい。

**投稿すべき意思決定**:
- 技術選定の理由（「なぜNext.jsではなくNuxtを選んだか」）
- 機能の優先順位づけ（「10個の要望から3つを選んだ基準」）
- 価格設定の根拠（「月額980円にした理由」）
- 却下した案とその理由

### 6. ユーザーの声を巻き込む

自分だけでなくユーザーを主役にすることで、コミュニティが生まれる。

- ユーザーのフィードバックをスクショ付きで紹介（許可を取って）
- 「次にどの機能を作るべき？」のアンケートを投稿
- ユーザーの使い方事例を紹介する
- ベータテスターを公開募集する

### 7. スレッド形式で深い知見を共有する

5〜10ツイートのスレッドは、1ツイートの5〜10倍のインプレッションを稼げる。

**効果的なスレッドの型**:
- 「◯◯を始めて△ヶ月。学んだことを全部書く」
- 「個人開発で月◯万円稼ぐまでにやった全リスト」
- 「◯◯の実装で3日ハマった話（解決方法付き）」

**スレッドの構成**:
1. フック（結論 or 驚きの事実）
2. 背景説明
3. 本題（3〜7ツイート）
4. まとめ + CTA（アプリリンク or フォロー呼びかけ）

### 8. 発信を効率化するテンプレートを作る

毎回ゼロから投稿内容を考えるのは非効率。テンプレートを用意する。

**週次テンプレート例**:
```
【進捗報告】
今週やったこと:
- [機能/改善1]
- [機能/改善2]
来週やること:
- [予定1]
- [予定2]
[スクリーンショット]
#個人開発
```

```
【学び共有】
[課題・問題]に直面した。
試したこと:
1. [アプローチ1] → ❌
2. [アプローチ2] → ❌
3. [アプローチ3] → ✅
結論: [学び]
#buildinpublic
```

## 実践事例

### 事例1: ビルドインパブリックでフォロワー0→5,000人（4ヶ月）
フリーランスエンジニアが副業で始めた請求書アプリの開発過程を毎日投稿。特に「今月の売上：¥0」「初めての有料ユーザー！」「解約された...原因を分析する」のリアルな投稿がバズり、4ヶ月でフォロワー5,000人。リリース初日に300DL。

### 事例2: 失敗投稿がバズって認知爆発
「3ヶ月かけて作ったアプリのローンチ初日、ダウンロード数は...3。何が間違っていたのか全部書く」というスレッドが2,000RT。その後のリカバリー過程も発信し続け、3ヶ月後には月間1,000DLに。失敗→復活のストーリーが強力なコンテンツに。

### 事例3: 数字公開で企業案件獲得
毎月の売上レポートを公開していた開発者が、テック系メディアから取材依頼。記事公開後にDL数が3倍に。さらに、数字の透明性を見た企業からAPI連携の提案が来て、B2B収益が追加された。

## すぐに試せるアクションアイテム

### 1. 週次投稿テンプレートを3つ用意する（30分）
- 月曜の「今週やること」テンプレート
- 金曜の「今週の振り返り」テンプレート
- 月次の「数字公開」テンプレート
- NotionやGitHub Gistに保存していつでもコピペできるようにする

### 2. 今日の作業をスクショ付きで投稿する（10分）
- 完璧でなくていい。途中のUIでもターミナルの画面でもOK
- 「今日はここまで進んだ」の一言 + スクショ + #個人開発
- 反応がなくても続ける。最初の1ヶ月は種まき期間

### 3. 直近1ヶ月の数字をまとめて公開する（30分）
- MRR、ユーザー数、PV等を1つの投稿にまとめる
- 「0」でも公開する（むしろ0からのスタートが一番共感される）
- 来月同じフォーマットで投稿して比較する
