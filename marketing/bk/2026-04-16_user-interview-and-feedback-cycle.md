# ユーザーインタビュー・フィードバック収集から改善サイクルを回す方法

## 概要

個人開発者が犯す最大の失敗は「自分が欲しい機能を作り続ける」こと。ユーザーの声を聞かずに開発すると、3ヶ月後に「誰も使わない機能だらけのアプリ」が出来上がる。逆に、5人のユーザーに30分ずつ話を聞くだけで、次に作るべき機能が明確になる。フィードバック収集は「余裕があったらやる」ものではなく、開発サイクルの中核に組み込むべきもの。

## 具体的な戦術

### 1. ユーザーインタビューの基本設計

**誰に聞くか（優先順）**:
1. 有料ユーザー（最も重要。お金を払う価値を感じている人）
2. 解約したユーザー（何が足りなかったかが分かる）
3. 無料のアクティブユーザー（有料化しない理由が分かる）
4. 登録したが使っていないユーザー（初期体験の問題が分かる）

**何人に聞くか**: 5人で十分。5人インタビューすれば重大な問題の85%が発見できる（Nielsen/Norman Groupの研究）。

**所要時間**: 1回20〜30分。月に2〜3回実施すれば十分。

### 2. インタビューで聞くべき質問（テンプレート）

**避けるべき質問**: 「この機能があったら使いますか？」（ほぼ全員「はい」と答えるが実際は使わない）

**効果的な質問リスト**:
1. 「このアプリを使う前、この作業をどうやっていましたか？」（過去の行動 = 真実）
2. 「最後にこのアプリを使ったとき、何をしましたか？」（具体的な利用シーン）
3. 「このアプリを使っていて一番イライラすることは？」（最大の改善ポイント）
4. 「もしこのアプリが明日なくなったら、何が困りますか？」（コアバリューの確認）
5. 「友人にこのアプリを説明するとしたら、なんて言いますか？」（ユーザーが感じる価値 = マーケティングコピーのヒント）
6. 「最近、他に試したアプリはありますか？」（競合と比較されるポイント）

### 3. アプリ内フィードバック収集の仕組み

インタビューは深い洞察が得られるが、スケールしない。アプリ内で自動的にフィードバックを集める仕組みも併用する。

**効果的な仕組み**:
- **NPS（Net Promoter Score）**: 月1回「このアプリを友人に勧めますか？（0〜10点）」を表示。9〜10点のユーザーはプロモーター、0〜6点はリスクユーザー
- **機能リクエストボード**: Canny、Nolt等のツールで「欲しい機能」に投票できる仕組み
- **アプリ内フィードバックボタン**: 全画面に「フィードバックを送る」ボタンを常設
- **解約時アンケート**: 解約理由を選択式 + 自由記述で収集

### 4. フィードバックの分類と優先順位付け

集めたフィードバックをそのまま実装リストにしてはいけない。分類と優先順位付けが必要。

**ICEスコアリング**:
- **Impact（影響度）**: この改善で何人のユーザーに影響するか（1〜10）
- **Confidence（確信度）**: この改善で本当に数字が動くか（1〜10）
- **Ease（容易さ）**: 実装にどれくらいかかるか（1〜10、簡単なほど高い）
- ICE = Impact × Confidence × Ease → スコアが高いものから着手

**分類の軸**:
| 分類 | 対応 | 例 |
|------|------|-----|
| バグ | 即修正 | 「保存できない」「クラッシュする」 |
| UX改善 | 次スプリント | 「分かりにくい」「手順が多い」 |
| 機能要望 | ICEスコアで判断 | 「◯◯機能が欲しい」 |
| 的外れ | 記録だけ | ターゲット外のユーザーからの要望 |

### 5. 「1人の声」と「多数の声」を区別する

1人の熱烈な要望に引きずられて機能を作ると、全体のバランスが崩れる。

**判断基準**:
- 同じ要望が3人以上から出ている → 優先度高
- 有料ユーザーからの要望 → 無料ユーザーの3倍の重み
- 解約理由として挙がっている → 最優先
- 「あったら嬉しい」レベル → 低優先（nice to have ≠ must have）

### 6. フィードバックループを2週間サイクルで回す

**Week 1**:
- 月〜水: フィードバック収集（アプリ内 + インタビュー1〜2件）
- 木〜金: 分類・優先順位付け

**Week 2**:
- 月〜木: 最優先の改善を実装・リリース
- 金: 改善結果の計測 + フィードバックをくれたユーザーに報告

**重要**: フィードバックをくれたユーザーに「あなたの声で◯◯を改善しました」と伝えることで、ロイヤルティが飛躍的に上がる。

### 7. 定量データと定性データを組み合わせる

フィードバック（定性）だけでは判断を誤ることがある。分析データ（定量）と組み合わせる。

- 「◯◯が使いにくい」（定性） + 「◯◯画面の離脱率が70%」（定量） = 確実に改善すべき
- 「◯◯機能が欲しい」（定性） + 「関連ページのPVが月10」（定量） = 需要は小さい
- Firebase Analytics、Mixpanel、PostHog等で行動データを計測

### 8. フィードバック疲れを防ぐ

ユーザーに聞きすぎると「またか」と離脱される。

- NPS/アンケートは月1回まで
- 同じユーザーに連続で表示しない
- インタビュー依頼はお礼（Amazonギフト券500〜1,000円 or アプリの1ヶ月無料）を用意
- フィードバックボタンは常設でOK（ユーザーが自発的に使う分には疲弊しない）

## 実践事例

### 事例1: 5人のインタビューで方向転換
メモアプリの開発者が有料ユーザー5人にインタビュー。「メモ機能よりもタスク連携が欲しい」が5人中4人の声。もともと予定していたリッチテキスト機能の開発を中止し、タスク連携を実装したところ、口コミで有料ユーザーが2ヶ月で3倍に。

### 事例2: 解約アンケートでチャーン50%削減
毎月の解約率が12%だったアプリが、解約フローにアンケートを追加。「使い方が分からない」が最多回答だったことを発見し、オンボーディングメールとチュートリアルを改善。3ヶ月後に解約率6%に。

### 事例3: 機能リクエストボードで開発効率2倍
Cannyで機能リクエストボードを公開。ユーザーが投票して上位に来た機能から実装する方針に変更。「作ったのに使われない」機能がゼロになり、リリースごとの利用率が平均40%→85%に向上。

## すぐに試せるアクションアイテム

### 1. 有料ユーザー3人にインタビューを申し込む（30分）
- アプリ内通知 or メールで「15分お話を聞かせてください」と依頼
- お礼として1ヶ月無料延長を提示
- 上記の質問テンプレートを手元に用意してZoomで実施

### 2. 解約フローにアンケートを追加する（2時間）
- 解約理由の選択肢: 「高い」「使わなくなった」「使い方が分からない」「別のツールに乗り換えた」「その他」
- 自由記述欄を追加
- 月次で集計して最多理由の改善に着手

### 3. フィードバック管理シートを作る（30分）
- スプレッドシートで「日付/ユーザー/フィードバック内容/分類/ICEスコア/対応状況」を管理
- 既存のフィードバック（X、レビュー、問い合わせ）を全て記入する
- 毎週金曜に15分でレビューして次週の開発方針を決める
